# v0.8 Completion Audit

Date: 2026-06-22

Status: complete for `v0.8 Server Backend Safety Foundation`.

This file is the final acceptance audit for v0.8. It answers one question:
is v0.8 done, and what evidence proves it?

Short answer: yes. v0.8 is complete. Real execution is still disabled.

## 1. Final Scope

v0.8 builds the safety foundation for a future controlled server backend. It
does not connect a real backend.

Completed v0.8 scope:

- server-backend safety design
- server job submission request model
- server job record model
- lifecycle state model
- approved job template model and validation
- workspace path validation
- in-memory lifecycle helpers
- test report
- Chinese learning summary
- study pack for later review

Explicitly not included:

- CUDA execution
- MPI execution
- SSH
- Slurm
- PBS
- remote server execution
- local wrapper execution
- credentials or private keys
- arbitrary shell execution from user text
- automatic Code Agent patch application

## 3. Runtime Safety Evidence

The most important runtime safety evidence is that non-`dry_run` backend values
remain rejected.

Code path:

```text
JobSubmissionRequest.backend_type
  -> validate_submission_boundary
  -> validate_backend_enabled
  -> only dry_run is accepted
```

Test evidence:

```text
ServerJobTest.RejectsNonDryRunSubmissionBeforeBackendsAreEnabled
```

Additional post-v0.8 preflight evidence:

```text
ServerJobTest.CompleteApprovalRecordDoesNotEnableRuntimeBackend
```

That later M11 preflight test proves that even a complete
`BackendApprovalDecision` record does not bypass `validate_backend_enabled`.

## 6. Interview Summary

Short pitch:

v0.8 adds the safety foundation for future lab job execution. It models future
submission requests, job records, approved templates, workspace guards, and
lifecycle events while keeping all real execution disabled.

Technical pitch:

The core design separates planning from execution. The Planner can produce
`ExperimentSpec` and dry-run `JobSpec`, but future submission must go through
`JobSubmissionRequest`, approved template validation, workspace validation, and
job record creation. Runtime backend validation still accepts only `dry_run`.

Safety pitch:

The version intentionally does not connect CUDA/MPI, SSH, Slurm, PBS, remote
servers, local wrappers, credentials, or arbitrary shell execution. The
important result is a tested safety boundary before any real backend exists.

## 7. What Is Still Not Allowed

Do not implement any of these as part of v0.8:

- run CUDA code
- run MPI code
- call `mpirun`
- call SSH
- call `sbatch`, `squeue`, `scancel`
- call `qsub`, `qstat`, `qdel`
- run a local wrapper script
- add credentials or private keys
- execute user-provided shell text
- let Code Agent auto-apply patches

These belong to a later milestone only after lab approval and operational
details are available.

## 8. Next Version Gate

The next milestone is M11, but M11 must not start real execution by default.

Before M11-T1 can continue, the lab must provide:
