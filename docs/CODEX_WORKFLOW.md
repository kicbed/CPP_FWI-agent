# Codex 自动接续与跨会话工作流

## 日常使用：直接提问

不需要记忆或执行任何项目专用启动脚本。在 Codex App、IDE 或 CLI 中正常打开本仓库，
然后直接提问即可。仓库根目录的 `AGENTS.md` 是自动入口：Codex 会在新会话开始时读取它，
并在处理第一项任务前自行完成以下动作：

1. 完整阅读短入口 `docs/PROJECT_CURRENT_STATE.md`；
2. 检查当前分支、工作树、最近提交和相关 diff，保护已有改动；
3. 按短入口路由，只读取当前阶段、当前 checkpoint、相关切片和相关 `D-*` 决策；
4. 只有阶段/范围变化、决策冲突、账本 reconciliation、安全/发布审计或维护持续记录本身时，
   才完整读取对应长文档；
5. 获取经过过滤的本机服务与 FWI 任务状态，或执行等价的安全只读检查；
6. 用现场 Git、交付物和受影响测试核对摘要，冲突时以现场证据为准并修正拥有该事实的真源；
7. 区分 Accepted、Implemented、Verified 和 Pending，再处理用户刚刚提出的问题。

用户只说“继续 D-003”时，默认执行一个有界工作轮次：先声明一个主要目标、一个风险边界、
预计触及面、验证层级和停止点，再实现并只跑受影响/失败测试。未到路线切片出口时以真实
`In progress` 安全交接；到切片出口才在候选最终 tree 上跑一次相关 aggregate；到阶段出口才
跑完整回归和代表性 CPU/CUDA/E2E。默认一次综合审阅，第三轮及以后独立审计须先获用户明确
批准。工作轮次不是新路线切片，不改变滚动余量。

官方行为说明见 [Codex 的 AGENTS.md 文档](https://learn.chatgpt.com/docs/agent-configuration/agents-md)。
项目级说明会从仓库根目录到当前工作目录逐级生效，因此在本仓库或其子目录中打开会话
都能获得根规则；更深目录如以后增加自己的 `AGENTS.md`，只覆盖其作用域内的规则。

`AGENTS.md` 在每个新会话启动时构建一次指令链。如果文件是在一个已经打开的会话中刚刚
修改的，重新开一个 Codex 会话即可让新版本生效；不需要运行项目脚本。

## 哪些内容会跨会话保留

跨会话记录分为四层，避免把决策、计划和短期进度混在一起：

- `docs/PROJECT_CURRENT_STATE.md` 保存不超过 80 行/8192 字节的读取路由、当前增量和不可破坏边界；
- `docs/PROJECT_CONTINUITY.md` 保存经用户明确同意的长期方向与安全边界；
- `docs/architecture/SCIENTIFIC_AGENT_RUNTIME_PLAN.md` 保存 D-003 的完整阶段、交付物和验收标准；
- `docs/PROJECT_PROGRESS.md` 是执行状态、验证证据、已验证 checkpoint、当前阶段、滚动余量和
  下一个安全动作的唯一真源。

`docs/PROJECT_CONTINUITY.md` 中的持久结论包括：

- 产品与架构方向；
- 安全边界和不得破坏的约束；
- 已实现、已验证和仍待实现的真实状态；
- 已知限制及可复现的验证依据。

用户说“保留这个”“以后都这样”或“记录下来”时，只授权把非 D 内容写入适当的现有文档，
不授权新增或修改编号 `D-*`。历史 D-001～D-012 保持原状。任何 D 的新增、删除、重编号、重排
或正文修改都是最高优先级受控操作：Codex 必须先展示只涉及一个 D 的完整精确 diff 及 SHA-256，
再给出一条填完整的 `D-AUTH` 授权句并停止；只有用户随后在同一对话中单独、原样复制该句才
单次有效。“同意”“继续”“固定”“记录”“修正”或认可方案内容都不能替代复制授权。没有有效
授权时，普通实现/验证状态只更新进度账本，不触碰 D 条目或其受保护 hash。

完整 D-003 的阶段是 P0→P1→P2→P3→P4→P5→P6；只有 P6 出口通过才算全项目完成，P5
不是终点。新发现的建议仍应先说明证据、收益、成本、风险和兼容性，但不得因此越过 D 锁。

这里不保存完整聊天记录、API Key、`.env` 内容、凭证、私有 prompt、模型数据或无必要的
个人信息。当前对话中的直接要求除最高优先级 D 锁外优先；直接要求改 D 只触发提案/hash，仍须
用户随后原样复制授权句。旧决定只有在该门通过后才能保留可追溯的 Superseded 记录。

临时 Codex/Claude 开发 prompt 与原始聊天在当前分支采用安全默认：不入 Git，放在
已忽略的 `.local-prompts/` 或 `*.local-prompt.md`。“决定产品 Agent 行为的脱敏运行时 prompt
应否作为可复现源码版本化”仍是 `D-005 Proposed`，不冒充用户已采纳结论。在用户
确认前，也不迁移或删除已有运行时 prompt。完整提案见 `docs/GIT_AND_PROMPT_POLICY.md`。

## 状态为什么仍需现场检查

持久文档记录的是决策，不是永远正确的运行快照。Git、测试、服务和 FWI job 会随时变化。
Codex 与开发进程看到同一个实时工作区，并会在相关操作前重新检查实际状态；项目不会启动
一个后台 watcher 持续读取目录，也不会把放进 `/root/fwi-runs` 的文件当成可执行任务。

## 内部诊断辅助程序

`scripts/codex-project.sh` 只保留给 Codex 自动状态刷新和开发者排障，**不是用户入口**。
`AGENTS.md` 要求 Codex 在需要时自行调用它的 `--print-context` 只读模式；辅助程序不可用时，
Codex 直接做等价检查并继续，不能要求用户代为执行。
辅助程序会验证短入口和四份持续性文档存在，并在启动摘要中提醒 Codex 按需深读；
它不把文档内容复制到 prompt，也不代替现场验证。

## CodeGraph 与 token 控制

本机已通过 MCP 提供 CodeGraph，仓库的 `.codegraph/` 索引由 Git 忽略。对架构、符号、调用链
和修改影响，Codex 应先做一次有界 CodeGraph 查询，避免宽泛 `rg` 加多次整文件读取。刚修改
或 pending-sync 的文件，以及未索引的 Markdown、Shell、SQL，仍直接定向读取；Git、测试、
运行状态和安全结论始终来自真实命令。CodeGraph 不可用时自动回退，不把初始化工作交给用户。

工作轮次内只运行受影响/失败测试并限制输出；路线切片出口只运行一次相关 aggregate，阶段出口
才运行完整回归和适用的代表性 CPU/CUDA/E2E。aggregate 只报告一次，已包含子集不重复相加；
数值 Worker、依赖、数据和规范化配置身份不变时复用绑定证据。这样减少重复执行与日志，不降低
阶段验收标准。

该程序的状态收集遵守以下边界：

- 不 source、解析或打印仓库 `.env`；
- 不读取日志、私有 prompt 或 FWI status 的自由文本；
- 只读取经校验的 `fwi-*` 状态文件白名单字段；
- 对敏感文件名过度过滤，不写入仓库缓存；
- 不发起 HTTP、SSH 或其他外部网络请求；
- 不启动后台 watcher，也不改变工作树。

它仍保留隔离测试，以防以后修改时破坏这些安全性质：

```bash
bash tests/test_codex_project_launcher.sh
```

该命令是维护者测试，不是日常使用步骤。测试使用临时 Git 仓库和假的 Codex 可执行文件，
不会联网或启动真实会话。
